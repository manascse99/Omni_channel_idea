1. System Overview 

OmniBank AI's backend is a central intelligence layer responsible for receiving customer 
messages from three channels — WhatsApp, Email, and Web Chat — and unifying them into a 
single, context-aware conversation thread per customer. No matter which channel a customer 
uses, all their messages are stored together, processed by AI, and presented to agents as one 
continuous timeline. 

Core Principle: One customer = One user profile = One conversation thread. Channel is just 
metadata, not identity. 

1.1 Three Pillars of the Backend

1.2 Key Design Decisions 

•  MongoDB is used for all storage — users, conversations, and messages — due to its 

flexibility with unstructured conversational data. 

•  AI processing runs in parallel with storage — the message is saved and the AI 

pipeline runs simultaneously, minimising response latency. 

•  The frontend is pre-built and fixed — the backend must expose APIs and Socket.io 

events that match exactly what the frontend expects. No frontend changes. 

•  Reply channel is AI-decided — based on the user's channel preference and history, 

the AI determines whether to reply via WhatsApp, Email, or Web Chat.

2.2 WhatsApp Webhook (Meta Cloud API) 

Meta sends a POST request to your registered webhook URL every time a customer message 
is received. The backend must: 

1.  Verify the webhook using the X-Hub-Signature-256 header (HMAC-SHA256 validation 

using your App Secret) 

2.  Parse the incoming payload to extract: phone number, message content, message type 

(text/media), timestamp, and WhatsApp message ID 

3.  Respond with HTTP 200 immediately — Meta will retry if no 200 is received within 20 

seconds 

4.  Pass the extracted data into the Central Processing Pipeline

12. If anonymous, create a temporary user profile that can be linked later when identity is 

confirmed

4. Conversation Management & Message Storage 

Once the user is identified or created, the backend manages the conversation layer. The design 
principle is strict: one unified conversation thread per user, regardless of channel. 

4.1 Conversation Lookup & Creation 

16. Query the conversations collection for an existing conversation where userId matches 

the resolved user's _id 

17. If a conversation exists — append the new message to that thread. Update the 

updatedAt, lastMessage, and status fields. 

18. If no conversation exists — create a new conversation document associated with that 

userId 

Design Rule: There is always exactly ONE conversation per user. New messages from any channel 
— WhatsApp, Email, or Web Chat — are always appended to this single thread. The channel is 
recorded per-message, not per-conversation.

5. AI Processing Pipeline 

AI processing runs in parallel with message storage — both operations are triggered 
simultaneously when a message arrives, so neither blocks the other. The AI pipeline uses 
Llama (running locally via Ollama) for all language processing tasks. 

Parallel Execution: Message storage and AI processing are fired simultaneously using Promise.all(). 
The AI response is sent back to the user only after both the message is stored AND the AI has 
generated a reply — typically within 1-3 seconds for local Ollama inference.

7. End-to-End Flow Walkthrough 

This section walks through a complete real-world scenario: a customer named Rajesh Kumar 
contacts the bank via WhatsApp asking about a personal loan, then follows up via email. 

Step 1 — WhatsApp message received 

•  Rajesh sends: 'I need a personal loan of ₹5,00,000' via WhatsApp to the bank number 
•  Meta Cloud API fires a POST to /webhook/whatsapp on the backend 
•  Backend validates signature, extracts phone: 919876543210 and message content 
•  Backend responds HTTP 200 to Meta immediately 

Step 2 — Identity resolution 

•  Backend queries MongoDB users for { phone: '919876543210' } → not found 
•  New user created: { _id: ObjectId('...'), phone: '919876543210', email: null, 

channelHistory: ['whatsapp'] } 

Step 3 — Conversation creation 

•  Backend queries conversations for { userId: Rajesh's _id } → not found 
•  New conversation created: { userId: ..., status: 'open', intent: null, sentiment: null } 

Step 4 — Message storage + AI processing (parallel) 

•  Message stored: { conversationId, channel: 'whatsapp', senderType: 'user', content: 'I 

need a personal loan...', isRead: false, timestamp: now } 

•  AI pipeline starts simultaneously — Llama processes the message 
•  AI returns: intent = 'loan_inquiry', sentiment = 'positive', suggestedReply = 'Hi Rajesh! 
Our personal loan rates start at 10.5% p.a. To check your eligibility, could you share 
your monthly income and preferred tenure?' 

Step 5 — AI reply sent 

•  Conversation updated: { status: 'ai-handling', intent: 'loan_inquiry', sentiment: 'positive', 

aiSummary: 'Customer inquiring about personal loan of ₹5L' } 

•  AI determines preferredChannel = 'whatsapp' (first contact channel) 
•  Backend sends reply via Meta Cloud API to Rajesh's WhatsApp 
•  AI reply stored: { senderType: 'ai', channel: 'whatsapp', content: 'Hi Rajesh! Our personal 

loan...' } 

•  Socket.io emits new_message and conversation_updated to agent dashboard

8.2 MongoDB Indexes (Required for Performance) 

conversations: index on userId, index on status, index on updatedAt (descending) 

•  users: unique index on phone (sparse), unique index on email (sparse) 
• 
•  messages: index on conversationId, index on timestamp (descending) 
•  messages: index on isRead for fast unread count queries 

8.3 Critical Rules 

•  Never block the webhook response — respond HTTP 200 to Meta/SendGrid 

immediately. All processing happens after the response. 

•  Deduplicate WhatsApp messages — check metadata.whatsappMsgId before storing. 

Meta may retry delivery. 

•  Never auto-merge duplicate profiles — always set duplicateWarning: true and let 

agent confirm. 

•  All API responses must match the shape the frontend expects — the frontend is 

fixed and cannot be changed. 

•  Socket.io rooms per conversation — use conversationId as the Socket.io room name 

so only agents in that conversation receive updates.
